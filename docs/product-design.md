# Westlake Now 产品设计方案

面向湖滨即时出行场景的 AI 本地路线规划产品

---

## 项目概述

**Westlake Now** 是一个面向杭州西湖湖滨周边“现在就出发”场景的 AI 本地路线规划产品。

用户只需要用自然语言描述当前需求，例如：

> 我在湖滨银泰，下午 2 点到 6 点，想逛西湖、喝咖啡、吃晚饭，人均 150，不想排队。
> 
> 

系统会自动识别用户输入中的出发地、时间、预算、偏好等约束，结合本地真实 POI 候选池、预算、地点类型、步行距离、等待风险和体验标签，生成一条当前可以执行的路线，并支持用户根据现场变化继续调整。

本项目的核心定位不是让 LLM 直接生成一篇旅游攻略，而是将 LLM 放在“理解用户意图和路线编辑意图”的位置；真正的路线选择、排序和局部再规划由后端 `routePlanner` 基于本地 `poiCatalog.json` 完成。前端负责将路线以移动端页面、时间轴、地点卡片、地图和调整差异的方式展示，并通过高德地图和高德导航完成从“生成路线”到“实际出发”的闭环。

当前版本聚焦杭州西湖湖滨周边即时出行，不做全城旅游规划、多日游攻略、实时排队承诺或复杂机器学习排序模型。

## 背景与问题

在真实出行和旅游场景中，用户经常不是提前完整规划，而是在某个地点附近临时决定“现在去哪儿”。

以西湖湖滨为例，用户可能刚到湖滨 in77，也可能刚下地铁、刚见到朋友、刚吃完饭，临时想在周边走一走、拍照、喝咖啡或吃晚餐。此时他们面临的问题不是“完全没有攻略”，而是攻略、地图、点评和导航之间没有形成一个即时决策闭环。

当前用户通常需要在多个工具之间来回切换：

1. 用地图搜索附近地点；

2. 用点评查看餐厅、咖啡店和排队情况；

3. 用攻略平台判断路线是否值得去；

4. 自己判断地点之间顺不顺路；

5. 自己估算时间、预算和步行距离；

6. 到现场发现排队、预算变化或时间不够时，再重新搜索和组合。

这会带来几个核心问题。

### 4.1 信息分散

地图、点评、攻略和导航分别提供不同信息，但用户最终需要的是一条可以直接执行的路线。传统工具提供信息，用户仍然需要自己筛选、组合和判断。

### 4.2 临时决策成本高

即时出行通常伴随多重约束：时间、预算、同行人、排队、天气、步行距离、拍照需求、是否想休息等。用户不是只想知道“哪里好玩”，而是想知道“我现在这样走是否合理”。

### 4.3 攻略内容不可直接执行

攻略通常是静态内容，很难根据“只剩 2 小时”“不想喝咖啡了”“餐厅排队太久”“预算降到 100”等现场变化进行局部调整。

### 4.4 纯 LLM 生成容易不可控

如果让大模型直接生成路线，容易出现地点不存在、路线顺序不合理、距离不准确、预算不可控、无法解释调整原因等问题。对于一个需要落地执行的出行产品来说，纯文本生成并不足够。

因此，本项目希望探索一种更可控的 AI 产品形态：让 LLM 理解自然语言，让本地 routePlanner 做路线决策，让真实 POI 数据提供可信候选池，让前端地图和导航完成执行闭环。

Westlake Now 的产品定位是：

> 面向湖滨即时出行的 AI 本地路线规划产品：将用户自然语言需求转化为可执行、可调整、可导航的本地路线方案。
> 
> 

它不是以下产品：

- 不是 AI 攻略生成器；

- 不是全城通用旅游规划工具；

- 不是多日游行程助手；

- 不是实时排队预测系统；

- 不是复杂机器学习推荐系统；

- 不是让 LLM 直接编写完整路线的聊天机器人。

它当前更像一个小范围、高确定性、可解释、可落地的路线决策系统。

核心分工是：

- **LLM / intent parser**：理解用户自然语言，将开放表达收敛成结构化约束或有限路线编辑操作；

- **routePlanner**：基于真实 POI 候选池做路线生成、排序和调整；

- **poiCatalog**：提供可控的本地 POI 数据底座；

- **前端**：展示路线、地图、地点卡片、时间轴、调整 diff 和导航入口；

- **高德能力**：用于 POI 数据来源、前端地图展示、定位和导航跳转。

## 移动端优先体验

Westlake Now 当前按照移动端优先的方式设计和实现。

这是因为“现在就出发”的路线决策场景通常发生在手机上：用户可能已经到达湖滨、正在路上、刚下地铁或和同行人临时讨论下一站。因此，产品体验重点不是复杂后台配置，而是让用户在手机上快速完成三件事：

1. 用一句话输入当前位置、时间、预算和偏好；

2. 快速查看当前路线、下一步行动和每个地点；

3. 在地图中确认路线，并通过“出发”跳转高德导航。

当前前端围绕移动端出行链路设计了首页输入、路线概览、下一步行动卡片、路线时间轴、地点详情、地图 Tab、底部导航操作栏和改路线入口。

### 8.1 首页输入页

首页强调“一句话，生成现在能走的路线”。用户可以直接输入自然语言需求，也可以使用当前位置或快捷需求标签。

### 8.2 路线时间轴页

路线结果页展示当前推荐、总时长、人均预算、步行距离、等待风险、下一步行动和路线时间轴。用户可以看到每一站的到达时间、离开时间、地点类型和下一段步行信息。

### 8.3 地图路线页

地图页展示路线节点和路线连线，并通过底部操作栏提供“改路线”和“出发”入口，使用户可以从路线规划自然过渡到高德导航。

## 核心用户流程

### 9.1 默认路线生成流程

1. 用户打开前端页面；

2. 用户输入自然语言需求，例如“我想现在出发逛西湖”；

3. 前端请求后端 `POST /api/chat-route`；

4. 后端解析用户输入；

5. intent parser / LLM 将用户需求转化为路线生成或调整意图；

6. routePlanner 基于 `poiCatalog.json` 选择路线节点；

7. 后端返回 `routeData`；

8. 前端展示约束 chips、路线概览、时间轴、地点卡片、地图和调整按钮；

9. 用户可以点击“出发”跳转高德导航。

### 9.2 现场调整流程

当用户在现场遇到变化时，不需要重新写一大段攻略需求，而是可以继续输入：

- “餐厅排队太久”

- “预算降到 100”

- “不想喝咖啡了”

- “只剩 2 小时”

- “想更适合拍照”

系统会将这些表达解析为有限的 `adjustmentType` 或 `constraintsPatch`，再由 routePlanner 对当前路线进行局部再规划。前端通过 `diff` 展示本次调整前后发生了什么变化。

## 核心功能设计

### 10.1 自然语言路线生成

用户可以用自然语言描述出行需求。系统会从中提取出发地、时间、预算、偏好、同行人、天气、是否少排队、是否适合拍照等约束。

当前后端请求体支持：

```JSON
{
  "message": "我想现在出发逛西湖"
}
```

实际接口也兼容更完整的上下文字段，例如：

```JSON
{
  "text": "",
  "message": "用户输入",
  "activeConstraints": {},
  "currentRoute": {},
  "currentPlaces": [],
  "currentConstraints": {},
  "routeData": {},
  "sessionId": ""
}
```

其中 `text` 和 `message` 都可作为输入文本，系统优先读取 `text`，否则读取 `message`。

### 10.2 本地 POI 候选池路线决策

当前路线生成不是由 LLM 直接编写，而是由后端 `routePlanner` 基于 `poiCatalog.json` 完成。

默认路线逻辑可以概括为：

```Plain Text
start → scenic → buffer → dinner
```

其中：

- `start`：默认起点，例如湖滨 in77；

- `scenic`：西湖湖滨周边景点；

- `buffer`：咖啡、商场、休息点等缓冲节点；

- `dinner`：晚餐节点。

routePlanner 会综合考虑 POI 类型、预算、停留时间、等待风险、拥挤风险、拍照分、聊天分、本地风味分、步行友好度等字段，并结合经纬度距离进行路线选择和排序。

### 10.3 五个高频调整场景

当前版本优先覆盖五个现场高频变化：

这五个调整不是简单按钮跳转，而是将用户现场变化转化为可执行的路线编辑操作。

### 10.4 route + diff 展示

系统不仅返回“当前路线”，还返回“本次调整发生了什么”。

- `route`：告诉用户当前路线是什么；

- `diff`：告诉用户这次调整改变了什么；

- `message`：用于解释系统建议；

- `adjustmentButtons`：给用户提供下一步可继续调整的方向。

例如用户输入“不想喝咖啡了”，系统会将原咖啡节点替换为非咖啡缓冲点，并在 `diff` 中展示：

- 原缓冲点；

- 新缓冲点；

- 预计人均变化；

- 总时长变化；

- 步行距离变化。

### 10.5 地图、定位与导航

前端已经实现：

1. 真实高德地图 Tab；

2. 高德定位能力；

3. 浏览器定位 fallback；

4. 距离过远提示；

5. 湖滨 in77 兜底起点；

6. “出发”跳转高德导航。

这让路线不只停留在文本层，而是可以被用户真正拿来执行。

## 自然语言理解边界与路线编辑操作

Westlake Now 当前不追求识别所有自然语言表达，而是通过 LLM / intent parser 将用户开放式输入收敛为有限、可执行的路线生成或路线编辑操作。

这种设计的核心原因是：路线规划不是纯文本生成任务，而是一个需要被前端执行、被地图展示、被用户导航使用的决策任务。如果直接追求“什么都能聊”，会增加结果不可控、字段不稳定和路线不可执行的风险。

当前系统优先覆盖高频路线编辑场景，包括：

1. **删除**
例如“不想喝咖啡了”，排除咖啡节点。

2. **替换**
例如“餐厅排队太久”，替换晚餐节点。

3. **增加**
例如“想更适合拍照”，增加或替换为拍照点。

4. **偏好调整**
例如“少排队”“想室内”“少走路”。

5. **停留时间调整**
例如“只剩 2 小时”，压缩路线。

6. **约束修改**
例如“预算降到 100”，重算预算约束下的路线。

对于无法明确解析的表达，后端应返回 clarification 或 fallback，而不是静默无反应。这样可以让用户知道系统需要补充什么信息，也能避免 LLM 解析失败直接破坏主流程。

## 系统架构

整体架构可以概括为：

```Plain Text

  ↓

  ↓
routeData
  ↓
```

### 12.1 前端层

前端负责：

- 用户输入；

- 路线结果展示；

- 时间轴展示；

- 地点卡片展示；

- 下一步行动卡片；

- diff 展示；

- 高德地图渲染；

- 使用当前位置；

- 距离过远提示；

- 高德导航跳转。

### 12.2 后端接口层

后端主要接口包括：

当前前端主链路使用：

```Plain Text
POST /api/chat-route
```

响应外层结构为：

```JSON
{
  "routeData": {}
}
```

### 12.3 意图解析层

意图解析层负责将自然语言转为结构化意图。

当前主逻辑是：

1. 优先尝试 LLM 解析；

2. 如果 LLM 不可用、失败或返回非法 JSON，则走规则 fallback；

3. 只输出有限类型的路线创建或路线调整意图；

4. 不让 LLM 直接生成路线、不编造 POI、不计算距离。

LLM 允许输出的核心结构包括：

```Plain Text
createRoute
adjustRoute
adjustmentType
constraintsPatch
```

支持的 `adjustmentType` 包括：

```Plain Text
restaurantBusy
budget100
noCoffee
twoHours
photo
null
```

### 12.4 路线决策层

`routePlanner` 是当前项目真正的路线决策核心，负责：

- 从 POI 候选池中选择节点；

- 根据类型组合路线；

- 根据预算和偏好筛选；

- 根据等待风险、拥挤风险和评分字段排序；

- 估算步行距离和步行时间；

- 生成时间轴；

- 输出路线概览；

- 处理五个高频调整场景；

- 输出 `routeData`。

### 12.5 数据层

`poiCatalog.json` 是当前路线决策的数据基础。

当前共有 25 个 POI，类型包括：

```JSON
{
  "start": 1,
  "scenic": 4,
  "photo": 4,
  "coffee": 4,
  "dinner": 6,
  "mall": 3,
  "rest": 1,
  "snack": 2
}
```

## LLM 与 routePlanner 的分工

本项目的一个核心设计原则是：

> LLM 负责理解，不负责路线决策；routePlanner 负责决策，不负责开放式语言理解。
> 
> 

### 13.1 LLM 负责什么

LLM / intent parser 负责：

1. 理解用户自然语言；

2. 判断用户是想创建路线还是调整路线；

3. 提取预算、时间、偏好、同行人等约束；

4. 将开放表达转化为有限 `adjustmentType`；

5. 在不确定时交给 fallback 或 clarification 机制。

### 13.2 LLM 不负责什么

LLM 不负责：

1. 直接生成完整路线；

2. 编造 POI；

3. 计算距离；

4. 判断真实营业状态；

5. 承诺排队情况；

6. 直接决定地点顺序。

这样可以降低大模型幻觉风险，避免路线结果不可控。

### 13.3 routePlanner 负责什么

routePlanner 负责：

1. 从 `poiCatalog.json` 中选择真实候选 POI；

2. 组合路线结构；

3. 根据规则评分选择节点；

4. 根据预算、时间、偏好调整路线；

5. 计算路线概览；

6. 生成 `timeline`；

7. 生成 `transportSegments`；

8. 生成调整后的 `diff`。

### 13.4 为什么这样分工

这种分工带来几个好处：

1. **可控性更强**
路线只会从本地候选池中选择，不会出现大模型编造地点。

2. **可解释性更强**
每次调整都可以通过 `diff` 告诉用户变了什么。

3. **工程落地更稳**
黑客松阶段不依赖复杂实时数据和模型训练，也能形成完整闭环。

4. **后续可扩展**
未来可以逐步替换距离估算、增加实时数据、扩展 POI 池，而不用重写产品架构。

## 路线生成与局部再规划逻辑

### 15.1 默认路线生成

默认路线会从本地 POI 候选池中选择一条包含起点、景点、缓冲点和晚餐点的路线。

示例默认路线：

```Plain Text
杭州湖滨in77
  → 杭州西湖风景名胜区-湖滨公园
  → EqualParts皮爷咖啡Peet's(杭州嘉里中心店)
  → 印湖杭味·创意杭帮菜(西湖店)
```

路线结果包括：

- 路线名称；

- 预计总时长；

- 预计人均预算；

- 预计步行距离；

- 等待风险；

- 地点顺序；

- 时间轴；

- 交通段。

### 15.2 局部再规划

当用户输入调整需求时，系统不会重新生成一篇攻略，而是基于当前路线做局部再规划。

例如：

- “不想喝咖啡” → 排除 `type=coffee` 的节点；

- “餐厅排队太久” → 替换晚餐节点；

- “预算降到100” → 选择更低成本的 POI 组合；

- “只剩2小时” → 删除或缩短缓冲节点；

- “想更适合拍照” → 引入拍照分更高的 POI。

### 15.3 路线距离与时间估算

当前 V1 使用经纬度直线距离估算步行时间。该方式适合黑客松阶段快速构建可用闭环，但不等同于真实道路级路径规划。

因此，文档和产品中需要明确：

> 当前步行时间为估算值，后续可接入高德步行路径规划进行替换。

## fallback / clarification / 容灾设计

当前产品并不追求识别所有自然语言表达，而是优先覆盖高频路线修改场景。

当 LLM 不可用、解析失败或返回非法结构时，系统会使用规则 fallback 继续解析高频意图，避免用户输入后系统静默无反应。

fallback 可覆盖的典型意图包括：

- 预算变化；

- 不想喝咖啡；

- 想少排队；

- 想拍照；

- 只剩 2 小时；

- 想休息；

- 想室内；

- 想少走路；

- 想吃正餐；

- 想吃小吃；

- 想经典西湖体验。

对于无法明确解析的表达，产品设计上应返回 clarification 或 fallback，而不是静默失败。

这种设计的意义是：

1. 降低 LLM 不稳定对主链路的影响；

2. 让用户知道系统需要什么信息；

3. 保证高频场景即使没有 LLM 也能继续工作；

4. 提升演示和真实体验的稳定性。

## 风险与应对

### 21.1 Render 冷启动

当前后端部署在 Render，可能存在冷启动延迟。

在服务已唤醒状态下，路线生成主要依赖本地候选池和规则引擎，具备较快响应能力；但冷启动情况下可能出现明显等待。

应对方式：

- 演示前预热后端服务；

- 提供健康检查地址；

- 后续可考虑常驻实例、缓存默认路线或更稳定的部署环境。

### 21.2 实时数据不足

当前不承诺实时排队和实时营业状态。

应对方式：

- 将 `waitRisk` 和 `crowdRisk` 定位为估计风险；

- 在文案中避免使用“保证不排队”“保证营业”等绝对表达；

- 后续可接入实时营业、排队或热力数据。

### 21.3 POI 数量有限

当前有 25 个 POI，适合湖滨小范围 MVP，但不足以支撑全城路线规划。

应对方式：

- 明确 MVP 范围；

- 后续扩展 POI 数量；

- 增加 POI 自动更新和质量校验机制。

### 21.4 LLM 解析不稳定

LLM 可能不可用、超时或返回不合规结构。

应对方式：

- 使用规则 fallback；

- 限制 LLM 输出结构；

- 不让 LLM 直接生成路线；

- 对不明确输入返回 clarification。

### 21.5 距离估算不够精细

当前 V1 使用经纬度直线距离估算步行时间，不能完全代表真实道路路径。

应对方式：

- 在路线说明中明确当前为估算；

- 后续接入高德步行路径规划；

- 将 `transportSummary` 作为透明说明字段。

### 23.1 从 AI 攻略生成转向 AI 路线决策

本项目没有让 LLM 直接写一篇攻略，而是将用户需求转化为可执行路线。

这使系统更接近真实产品，而不是单纯文本生成 Demo。

### 23.2 LLM 与 routePlanner 分工清晰

LLM 负责理解自然语言，routePlanner 负责路线决策。

这种架构降低了大模型幻觉，提升了系统可控性。

### 23.3 使用真实 POI 数据

当前 25 个 POI 基于高德来源信息整理，包含经纬度、地址、营业时间、人均成本、标签、风险和评分字段。

这使路线生成不是凭空编造，而是基于真实候选池。

### 23.4 支持现场局部再规划

用户不是只能生成一次路线，而是可以根据现场变化继续调整。
`route + diff` 设计让用户知道路线为什么变化、变了哪里。

### 23.5 完整执行闭环

当前系统已完成：

```Plain Text
自然语言输入
→ 意图解析
→ 路线生成
→ 路线调整
→ 地图展示
→ 当前位置
→ 导航跳转
```

这说明项目不是停留在概念层，而是具备真实可体验的产品闭环。

### 23.6 移动端即时出行体验

产品从一开始按照手机使用场景设计，而不是把桌面端信息压缩到移动端。首页输入、下一步行动、时间轴、地图和底部操作栏都围绕“现场快速决策和出发”展开。

## 团队分工

本项目由两人协作完成，采用产品定义、前端实现、后端路线规划、数据整理、联调验收并行推进的方式。

### 24.1 产品与前端方向

负责内容包括：

1. 项目选题与产品定位；

2. 用户场景拆解与 MVP 范围定义；

3. 产品需求文档、路线规则文档和产品设计方案整理；

4. 前端页面结构、移动端交互和视觉体验设计；

5. 首页输入、路线结果、地点卡片、时间轴、地图 Tab、底部操作栏等前端能力实现；

6. 前后端字段契约协作，包括 `routeData`、`route.placeIds`、`diff`、`adjustmentButtons` 等核心字段对齐；

7. 真实 POI 候选池的产品侧定义与字段使用设计；

8. 部署、验收、Demo 链路和提交材料整理；

9. 项目后期补位处理部分后端联调、修复和验收工作。

### 24.2 后端方向

负责内容包括：

1. 后端服务搭建；

2. 路线生成接口实现；

3. `routePlanner` 核心规则逻辑；

4. `poiCatalog.json` 接入；

5. LLM / fallback 意图解析链路；

6. 五个核心调整场景的后端实现；

7. 后端部署和接口联调支持。

### 24.3 协作方式

项目采用前后端字段契约驱动的协作方式推进。前端不直接假设后端返回内容，而是围绕 `routeData` 中的 `constraints / places / route / diff / message / adjustmentButtons` 进行展示；后端则围绕前端可执行、可展示、可调整的需求组织接口结构。

这种协作方式保证了项目在黑客松时间限制下仍然能形成完整闭环：从产品定义、真实数据、后端规则、自然语言解析、前端展示到线上部署。

## 评审关注点对应

### 25.1 完整性

项目已经完成从输入到执行的核心闭环：

1. 前端已上线；

2. 后端已部署；

3. 主接口 `/api/chat-route` 已完成；

4. 25 个真实 POI 已整理为本地候选池；

5. routePlanner 已完成默认路线生成；

6. 已支持五个高频调整场景；

7. 前端已完成路线时间轴、地点卡片、下一步行动、地图 Tab 和底部操作栏；

8. 已支持使用当前位置；

9. 已支持跳转高德导航；

10. 已通过 `routeData` 字段契约完成前后端联调。

### 25.2 创新性

项目的创新点不只是“使用 LLM”，而是重新定义 LLM 在路线规划产品中的位置：

1. 不让 LLM 直接生成攻略；

2. 让 LLM 负责理解自然语言和编辑意图；

3. 让 routePlanner 负责真实路线决策；

4. 用本地 `poiCatalog` 控制候选地点范围；

5. 用 `route + diff` 解释路线变化；

6. 用移动端地图和导航承接真实执行。

这种设计兼顾了 AI 交互体验和工程可控性。

### 25.3 应用效果

用户可以完成完整使用链路：

```Plain Text
输入一句话需求
→ 系统识别时间 / 预算 / 出发地 / 偏好
→ 生成当前可执行路线
→ 查看路线概览和时间轴
→ 查看地图路线
→ 根据现场变化调整路线
→ 点击“出发”跳转高德导航
```

相比静态攻略，Westlake Now 更适合“现在就出发”的现场决策场景。

### 25.4 工程可维护性

项目通过字段契约和模块分工降低维护成本：

1. `routeData` 字段结构稳定；

2. 前端按 `route.placeIds` 渲染路线顺序；

3. `places` 作为地点详情列表，和当前路线顺序解耦；

4. LLM 只输出有限结构，避免直接污染路线结果；

5. routePlanner 独立承担路线决策，便于后续替换距离计算、扩展 POI 或接入实时数据；

6. fallback 机制降低 LLM 不稳定对主链路的影响。

### 25.5 边界诚实

当前版本明确不夸大能力：

1. 不承诺全城规划；

2. 不承诺多日游；

3. 不承诺实时排队；

4. 不承诺实时人流；

5. 不承诺绝对营业；

6. 不把高德实时路径规划作为主链路；

7. 不声称支持全量自然语言理解；

8. 不声称使用复杂机器学习排序模型。

这种边界控制保证了项目描述和实际实现一致，也为后续迭代留下清晰空间。

## 总结

Westlake Now 的核心价值不是“生成一份更长的攻略”，而是帮助用户在湖滨周边即时出行时，快速得到一条能走、能看、能调、能导航的本地路线。

当前版本通过 LLM 意图解析、routePlanner 路线决策、真实 POI 候选池、`route + diff` 展示、高德地图和导航跳转，完成了从用户自然语言输入到真实路线执行的闭环。

在黑客松阶段，项目选择了克制的 MVP 范围：不做全城、不做多日游、不承诺实时排队、不让 LLM 直接编造路线，而是优先保证小范围内的可执行性、可解释性和演示稳定性。

这也是 Westlake Now 的产品设计核心：

**用 AI 降低即时出行决策成本，用可控路线规划保证结果可以真正出发。**
