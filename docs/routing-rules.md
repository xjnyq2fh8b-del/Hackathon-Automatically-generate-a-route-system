## 文档目标

本文档用于定义本项目第一版“路线规划规则 / 算法规则”，帮助后端同学和 Codex 根据产品规则实现后端路线决策逻辑。

本项目不是让 LLM 直接生成一篇旅游攻略，而是要做一个能落地的本地路线决策系统。

第一版要解决的问题是：

> 用户用自然语言表达当前位置、时间、预算、同行人群和偏好后，系统能将需求转化为结构化约束，结合真实 POI、距离、营业时间、自建体验标签和等待风险估计，生成一条现在可以直接执行的路线，并在用户现场改变需求时进行局部再规划。

文档核心目标：

1. 明确默认路线如何生成。
2. 明确 POI 如何评分和排序。
3. 明确硬约束、风险约束、软偏好的区别。
4. 明确策略化调整如何影响路线。
5. 明确局部再规划如何做最小必要改动。
6. 明确 LLM、高德 API、后端规则引擎、前端分别负责什么。
7. 明确 V1 做什么、不做什么，以及后续如何演进。

## 总体设计原则

### 4.1 第一优先级：稳妥可执行

默认路线的第一目标不是惊喜，也不是地点最多，而是稳妥可执行。

路线必须满足：

1. 地点真实存在；
2. 顺序合理；
3. 不明显绕路；
4. 不明显超时；
5. 不明显超预算；
6. 不包含用户明确排除的节点；
7. 风险可解释；
8. 用户拿到后能现在就走。

---

### 4.2 第二优先级：体验完整

默认路线追求的是体验完整，不是体验最丰富。

体验完整的含义是：

> 在时间、预算和步行可控的前提下，形成一个有节奏的路线闭环。

当前西湖场景下，体验完整通常表现为：

```text
起点 → 核心体验点 → 休息 / 缓冲 / 拍照增强点 → 餐饮点
```

但不是每条路线都必须包含咖啡。咖啡只是中段缓冲的一种实现方式。

### 4.4 第四优先级：亮点 / 记忆点

亮点不是随机惊喜，也不是 LLM 编出来的小红书文案。

第一版亮点来自自建体验标签，例如：

- 湖景；
- 拍照友好；
- 西湖经典；
- 不太网红；
- 安静；
- 适合聊天；
- 家庭友好；
- 预算友好；
- 可休息。

亮点是加分项，不能压过路线可执行性。

## 系统分工

### 5.1 为什么需要明确系统分工

本项目不是让大模型直接生成一篇旅游攻略，而是要做一个能落地的本地路线决策系统。

路线推荐涉及：

- 真实地点；
- 位置距离；
- 营业时间；
- 预算；
- 等待风险；
- 路线顺序；
- 用户偏好；
- 现场调整；
- 新旧方案差异。

如果所有决策都交给 LLM 直接生成，容易出现：

1. 编造不存在或不准确的 POI；
2. 距离、时间和营业状态不可靠；
3. 预算、人均和等待风险不可验证；
4. 多个地点串联后可能绕路或超时；
5. 用户调整后无法稳定解释新旧变化；
6. 结果看起来像攻略文案，但不一定能执行。

因此，第一版采用以下分工：

```text
LLM 负责理解和表达
高德 API 负责真实地理与 POI 数据
后端规则引擎负责路线决策
前端负责展示和交互
```

---

### 5.2 LLM 负责什么

LLM 主要负责“理解用户”和“解释结果”，不负责最终路线决策。

LLM 负责：

1. 解析自然语言输入；
2. 识别用户意图；
3. 抽取结构化约束；
4. 判断用户是在新增、删除、替换、缩短时间、降低预算，还是改变偏好；
5. 生成推荐理由；
6. 生成新旧方案变化解释；
7. 把规则引擎结果转成用户能理解的自然语言。

示例：

用户输入：

```text
我在湖滨银泰，下午 2 点到 6 点，想逛西湖、喝咖啡、吃晚饭，人均 150，不想排队。
```

LLM 应解析为：

```JSON
{
  "origin": "湖滨银泰 in77",
  "timeWindow": "14:00-18:00",
  "budgetMax": 150,
  "preferences": ["逛西湖", "喝咖啡", "吃晚饭", "少排队"],
  "companions": null,
  "avoid": []
}
```

用户输入：

```text
餐厅排队太久，换个不用等太久的。
```

LLM 应解析为：

```JSON
{
  "adjustmentType": "restaurantBusy",
  "targetNodeType": "dinner",
  "riskConstraint": {
    "waitRisk": "lower"
  }
}
```

LLM 不应该直接输出完整路线，也不应该自行编造地点、人均、营业时间和步行距离。

---

### 5.3 高德 API 负责什么

高德 API 主要负责提供真实地理数据和 POI 基础信息。

高德负责：

1. POI 搜索；
2. 地点名称；
3. 地址；
4. 经纬度；
5. POI 类型；
6. 距离；
7. 地图点位展示；
8. 路径距离或通勤时间估算；
9. 部分营业时间和评分信息，视 API 返回能力而定。

高德不负责：

1. 判断一个地点是否适合朋友聊天；
2. 判断一个地点是否适合拍照；
3. 判断一个地点是否适合爸妈；
4. 判断一个地点是否“游客太多”；
5. 判断一条路线是否体验完整；
6. 判断用户在预算、时间、距离、拍照之间应该怎么取舍。

高德给的是事实基础，路线选择和排序仍由后端规则引擎完成。

---

### 5.4 后端规则引擎负责什么

后端规则引擎是路线决策核心。

后端负责：

1. 接收结构化约束；
2. 读取高德 POI 和本地 `poiCatalog`；
3. 进行硬约束过滤；
4. 进行风险约束降权；
5. 进行软偏好加权打分；
6. 组合候选 POI 成路线；
7. 计算路线总时长、预算、步行距离和等待风险；
8. 生成默认路线；
9. 处理局部再规划；
10. 生成新旧方案 diff；
11. 将内部算法数据转换为前端可展示结构。

后端规则引擎不应该只是把用户问题转发给 LLM。

第一版后端可以先使用固定权重和规则表实现，不需要复杂机器学习模型。

---

### 5.5 前端负责什么

前端负责展示和交互，不负责路线算法。

前端负责：

1. 展示用户约束摘要；
2. 展示路线总览；
3. 展示地点卡片；
4. 展示时间轴；
5. 展示交通段；
6. 展示调整按钮；
7. 展示新旧方案 diff；
8. 支持地点卡片删除、替换、上移、下移等操作；
9. 将用户操作传给后端；
10. 展示后端返回的新路线。

前端不判断哪条路线更优，也不在本地写死路线调整逻辑。

```text

route

```

其中：

调整接口第一版返回：

```text
route + diff
```

不要返回 `routePatch`。

---

### 6.2 `diff.rows` 格式统一

当前 `diff.rows` 统一使用对象数组：

```JSON
{
  "title": "已避开排队晚餐",
  "action": "晚餐改为等待风险更低的餐厅，其他节点保持不变。",
  "rows": [
    { "label": "晚餐", "value": "新白鹿餐厅湖滨店 → 弄堂里湖滨店" },
    { "label": "等待风险", "value": "低-中 → 低" },
    { "label": "预计人均", "value": "120元 → 108元" }
  ]
}
```

不建议再使用二维数组：

```JSON
[
  ["预计人均", "120元 → 108元"]
]
```

原因：

1. `\{ label, value \}` 可读性更强；
2. 不容易写反顺序；
3. 以后方便扩展字段；
4. 当前 mock data 已经是对象数组；
5. 前端展示逻辑更稳定。

---

### 6.3 ⭐ 前端展示字段与后端算法字段分离

第一版需要区分两类字段：

1. 前端展示字段；
2. 后端内部算法字段。

前端展示字段包括：

```text
constraints
places
route
diff
```

这些字段用于页面展示，不宜在联调阶段频繁修改。

后端内部算法字段包括：

```text
poiCatalog
experienceTags
avgCost
stayMinutes
waitRisk
crowdRisk
photoScore
quietScore
restScore
familyScore
chatScore
classicScore
budgetScore
walkFriendlyScore
localFlavorScore
```

这些字段用于评分、筛选和排序，不一定全部返回给前端。

其中：

- `tags`：前端展示用中文标签，例如“湖景”“拍照友好”“可休息”；
- `experienceTags`：后端内部算法标签，例如 `photo`、`lake-view`、`classic`、`low-crowd`、`local-cuisine`、`quiet`；
- score 字段：用于表示某个体验维度的强弱，例如 `photoScore = 5` 表示拍照友好度很高。

规则：

1. 前端只关心最终选中的路线；
2. 后端可以维护更丰富的候选池；
3. 算法字段不要直接污染前端展示结构；
4. 如果需要真实地图展示，可以在 `places` 中向后兼容增加 `location: \{ lng, lat \}`；
5. 不要删除现有 `map` 字段，避免影响当前页面展示；
6. `tags` 面向用户阅读，`experienceTags` 面向规则引擎筛选和排序。

## ⭐ POI 数据来源与候选池

### 7.1 数据来源三层结构

第一版 POI 数据来源分为三层：

```text
高德 POI 搜索
  ↓
后端 poiCatalog 候选池
  ↓
规则引擎评分排序
  ↓
前端 places + route + diff 展示
```

---

### 7.2 高德 POI 搜索

高德 API 用于获取真实地点基础信息，包括：

- POI 名称；
- 地址；
- 经纬度；
- 类型；
- 距离；
- 评分或热度信息，视 API 返回能力而定；
- 营业时间，视 API 返回能力而定。

高德负责提供真实地理数据，但不负责判断地点是否适合当前用户体验。

例如，高德可以告诉系统“附近有哪些咖啡店”，但不能直接判断：

- 哪家更适合朋友聊天；
- 哪个湖景点更出片；
- 哪个餐厅更适合爸妈；
- 哪个地点游客更少；
- 哪条路线整体更舒服。

这些由后端规则引擎和自建体验标签负责。

---

### 7.3 后端 `poiCatalog` 候选池

后端需要将高德返回的 POI 标准化为内部候选池。

`poiCatalog` 是后端内部算法数据，不要求前端完整展示。

建议字段如下：

```JSON
{
  "id": "poi_001",
  "amapId": "高德POI ID",
  "name": "地点名称",
  "type": "scenic | coffee | dinner | rest | snack | mall",
  "address": "地址",
  "location": {
    "lng": 120.000000,
    "lat": 30.000000
  },
  "openHours": "10:00-21:30",
  "avgCost": 42,
  "rating": 4.5,
  "source": "amap",

  "tags": ["湖景", "拍照友好", "可休息"],
  "experienceTags": ["lake-view", "photo", "rest-friendly"],

  "stayMinutes": 30,
  "waitRisk": "low | medium | high",
  "crowdRisk": "low | medium | high",

  "photoScore": 1,
  "quietScore": 1,
  "restScore": 1,
  "familyScore": 1,
  "chatScore": 1,
  "classicScore": 1,
  "localFlavorScore": 1,
  "budgetScore": 1,
  "walkFriendlyScore": 1
}
```

字段说明：

- `amapId` 用于追溯高德来源；
- `location` 用于地图展示和距离计算；
- `avgCost` 用于预算计算；
- `stayMinutes` 用于路线时间轴；
- `tags` 是前端展示用中文标签；
- `experienceTags` 是后端内部算法标签，用于候选筛选；
- `waitRisk` 和 `crowdRisk` 属于风险估计，不代表实时精确数据；
- 各类 `Score` 字段用于后端排序，不必全部展示给前端。

示例理解：

```JSON
{
  "type": "scenic",
  "tags": ["湖景", "拍照友好", "西湖经典"],
  "experienceTags": ["lake-view", "photo", "classic"],
  "photoScore": 5,
  "classicScore": 4,
  "crowdRisk": "medium"
}
```

前端主要展示 `tags`。
后端算法主要使用 `type + experienceTags + score`。

---

### 7.4 15-25 个真实 POI 候选池

第一版不建议只靠当前 7 个 mock POI 支撑完整算法叙事。

建议扩展一个小范围真实 POI 候选池，规模控制在 15-25 个。

这些 POI 不需要全部展示给前端，但需要供后端规则引擎选择。

建议类型分布：

推荐做法：

```text
高德 API 拉真实 POI
  ↓
筛选西湖湖滨周边
  ↓
人工校验少量核心点
  ↓
补充自建体验标签
  ↓
生成 poiCatalog.json
  ↓
后端规则引擎评分排序
```

---

### 7.5 人工标签怎么标注

不要只做“拍照高 / 中 / 低”这种单一标签。

更合理的是：每个 POI 有一组 1-5 分的体验画像。

建议评分字段：

建议风险字段：

建议体验标签字段：

```JSON
{
  "experienceTags": ["photo", "lake-view", "classic", "low-crowd", "local-cuisine", "quiet"]
}
```

说明：

- `experienceTags` 是后端内部体验标签，用于候选筛选；
- score 字段用于表示某个体验维度的强弱；
- risk 字段用于表示等待、拥挤等不确定风险；
- 前端展示用的中文 `tags` 可以继续保留，例如“湖景”“拍照友好”“可休息”；
- `experienceTags` 不一定直接展示给用户。

示例：

```JSON
{
  "type": "scenic",
  "tags": ["湖景", "拍照友好", "西湖经典"],
  "experienceTags": ["lake-view", "photo", "classic"],
  "photoScore": 5,
  "classicScore": 4,
  "quietScore": 3,
  "crowdRisk": "medium",
  "waitRisk": "low"
}
```

这表示该 POI 适合作为湖景 / 拍照 / 西湖经典候选点，但是否最终被选中，还要结合距离、时间、预算、拥挤风险和当前路线结构一起判断。

---

### 7.6 防止重复推荐

为了避免不同策略都推荐同一个 POI，规则引擎需要增加候选多样性控制。

建议规则：

1. 每个类型至少保留 Top 3 候选，不直接只取最高分；
2. 同一条路线中，不重复放入功能相同且距离过近的节点；
3. 拍照、休息、餐饮、核心体验点应尽量承担不同角色；
4. 当两个 POI 分数接近时，优先选择更顺路、风险更低、解释更清楚的点；
5. 如果某个 POI 已经在当前路线中，局部再规划时除非必要，不重复把它作为新增点；
6. 策略调整优先替换受影响节点，不整条路线重算。

## ⭐ 约束分类：硬约束、风险约束、软偏好

### 8.1 为什么不只分硬约束和软偏好

真实用户不会总是说标准句。

用户输入往往是混乱的，例如：


## ⭐ POI 评分与推荐排序规则

### 9.1 设计目标

POI 评分与推荐排序用于解决两个问题：

1. 在多个候选地点中，选出更适合当前用户约束和路线结构的地点；
2. 在用户提出调整需求时，优先替换受影响节点，而不是重新生成一条完全不同的路线。

第一版评分规则采用：

```text
硬约束过滤 + 风险约束降权 + 软偏好加权排序
```

本项目第一版不训练复杂机器学习模型，而是采用可解释的规则算法。

原因：

1. MVP 阶段缺少真实用户行为数据；
2. 缺少路线满意度标注；
3. 规则算法更稳定；
4. 更可控；
5. 更可解释；
6. 更适合黑客松周期内落地。

---

### 9.2 评分总体流程

第一版评分流程如下：

```text
输入用户约束
  ↓
生成候选 POI 池
  ↓
硬约束过滤
  ↓
风险约束降权
  ↓
软偏好加权排序
  ↓
组合候选路线
  ↓
计算路线总分
  ↓
输出最佳路线
```

核心原则：

1. 硬约束先过滤，不进入打分；
2. 风险约束不轻易承诺绝对满足，只做排除高风险和降权处理；
3. 软偏好用于排序，不应压过时间、距离和预算等可执行性指标；
4. 评分只在可行候选中比较，不用于给不可行路线“洗白”。

---

### 9.3 硬约束过滤

硬约束是必须满足的条件。

不满足硬约束的 POI 或路线不能进入主推荐路线。

示例：

用户说“不要咖啡”，系统必须删除或过滤咖啡节点，不能继续推荐 coffee 类型 POI。

用户说“只剩 2 小时”，系统不能生成 2 小时 30 分钟的主路线。

---

### 9.4 风险约束处理

风险约束是用户强烈在意，但系统无法实时、绝对保证的条件。

例如：

- 不想排队；
- 不要游客太多；
- 带爸妈，别太累；
- 想安静一点。

这些需求不能简单承诺“绝对不排队”或“绝对人少”，因为第一版没有实时排队和实时人流数据。

处理方式：

风险提示文案必须克制：

- 可以说“等待风险较低”；
- 可以说“游客相对分散”；
- 不应说“保证不排队”；
- 不应说“实时人少”。

---

### 9.5 软偏好加权排序

软偏好是尽量满足的体验需求，可以在必要时为了时间、距离、预算做取舍。

例如：

- 想拍照；
- 想吃杭帮菜；
- 朋友聊天；
- 想要本地特色；
- 想更有湖景；
- 想轻松一点。

软偏好通过调整评分权重影响排序，而不是直接过滤所有不匹配地点。

---

### 9.6 ⭐ 默认评分维度与权重

默认路线的第一目标是稳妥可执行，第二目标是体验完整，第三目标才是亮点和惊喜。

第一版默认权重如下：

合计 100 分。

---

### 9.7 权重依据

这套权重不是拍脑门，而是冷启动启发式权重。

依据如下：

#### 9.7.1 错误成本

如果时间错了、距离太远、路线不顺，用户直接用不了。

所以：

- 时间可执行性最高；
- 距离 / 顺路程度第二；
- 体验完整度保持较高。

如果亮点少一点，路线仍然可以用。

所以亮点 / 记忆点权重最低。

---

#### 9.7.2 数据可信度

时间、距离、预算相对更可计算。

拍照、人少、安静、低等待在 V1 阶段更多依赖标签估计，可信度低一些。

所以不能把“亮点”“安静”“出片”权重设得过高。

---

#### 9.7.3 产品定位

本项目是“现在就出发”的即时路线决策，不是小红书攻略生成器。

所以默认路线要先稳，再个性化。

---

#### 9.7.4 交付要求

第一版要证明：

1. 路线可执行；
2. 多约束能处理；
3. 调整后结果真实变化；
4. 新旧方案能对比；
5. 不是纯 LLM 文案。

因此可执行性和完整性优先。

---

### 9.8 ⭐ 单个 POI 得分方式

后端可以先采用简单加权计算：

```text
POI 得分 =
时间可执行性分 * 25%
+ 距离 / 顺路分 * 20%
+ 体验完整度分 * 18%
+ 预算匹配分 * 14%
+ 等待风险分 * 14%
+ 偏好匹配分 * 6%
+ 亮点记忆点分 * 3%
```

每个维度可以按 1-5 分计算。

注意：如果某个 POI 已经违反硬约束，不参与打分。

---

路线不能只看单点得分，还要看组合效果。

路线总分建议由以下部分组成：

路线总分不是简单把 POI 得分相加。

如果单个 POI 分数很高，但加入后导致路线绕路、超时或节奏变差，该 POI 不应被选入最终路线。

## ⭐ 路线生成与局部再规划规则

### 11.1 设计目标

路线生成模块解决的问题不是“列出几个推荐地点”，而是把多个 POI 组织成一条用户现在能直接执行的路线。

第一版路线生成目标：

1. 生成一条稳妥可执行的默认路线；
2. 保证路线包含明确顺序、时间轴、交通段和总指标；
3. 支持用户现场变化后的局部再规划；
4. 每次调整都能解释“改了什么、为什么改、牺牲了什么”；
5. 避免让 LLM 直接编完整路线，路线决策由后端规则引擎完成。

第一版不追求：

1. 全城最优路线；
2. 复杂多日行程；
3. 全量实时交通最优解；
4. 机器学习自动生成路线；
5. 无限多备选方案。

---

### 11.2 默认路线生成原则

默认路线的优先级为：

```text
稳妥可执行 > 体验完整 > 路线顺路 > 风险可控 > 个性化亮点
```

默认路线不是景点越多越好，也不是越便宜越好，而是要让用户在当前时间、预算和区域内马上可以执行。

默认路线必须满足：

1. 地点真实存在；
2. 节点顺序合理；
3. 时间不明显超出用户可用时间；
4. 总步行距离可接受；
5. 不包含用户明确排除的类型；
6. 不安排明显高风险节点；
7. 能形成基本体验闭环。

---

### 11.3 默认路线骨架

第一版默认路线采用以下骨架：

```text
起点 → 核心体验点 → 中段缓冲点 → 餐饮点
```

对应当前 demo：

```text
湖滨银泰 in77 → 断桥残雪 → 湖畔白塔咖啡 → 新白鹿餐厅湖滨店
```

各节点职责如下：

---

### 11.4 节点类型定义

#### 11.4.1 起点节点

起点节点来自用户当前位置或用户输入的出发地。

规则：

1. 起点默认不可删除；
2. 如果用户修改出发地，需要整条路线重新计算；
3. 起点不计入消费预算；
4. 起点影响后续所有 POI 的距离和顺路判断。

---

#### 11.4.2 核心体验点

核心体验点用于保证路线有明确的西湖体验。

核心体验点不一定等于热门景点，也可以是湖景点、安静观景点、拍照点或低拥挤体验点。

规则：

1. 用户第一次来杭州、明确想逛西湖时，必须包含核心体验点；
2. 用户时间紧时，核心体验点优先保留，缓冲点优先删除；
3. 用户想安静时，不取消西湖体验，而是把热门核心点替换为低拥挤湖景点；
4. 用户想拍照时，核心体验点可以同时承担拍照点功能；
5. 如果核心体验点拥挤风险过高，应寻找附近替代点，而不是直接删除西湖体验。

---

#### 11.4.3 中段缓冲点

中段缓冲点用于改善路线节奏。

它可以是：

1. 咖啡店；
2. 茶饮店；
3. 轻休息点；
4. 小吃点；
5. 湖边停留点；
6. 商场休息点；
7. 拍照增强点。

规则：

1. 用户明确说喝咖啡时，咖啡节点必须有；
2. 用户明确说不要咖啡时，咖啡节点必须删除或替换；
3. 用户带爸妈、朋友聊天、别太累时，需要有“可休息能力”，但不一定必须是咖啡；
4. 用户只剩 2 小时时，中段缓冲点优先删除；
5. 用户预算较低时，咖啡节点可以替换为轻休息点或小吃点；
6. 用户想拍照时，中段缓冲点可以替换为拍照增强点。

---

#### 11.4.4 餐饮点

餐饮点用于覆盖饭点和路线收尾。

规则：

1. 用户明确说吃饭、晚饭、餐厅时，餐饮点必须有；
2. 用户时间覆盖饭点时，餐饮点强推荐；
3. 用户明确说不吃饭时，餐饮点必须删除；
4. 用户预算较低时，正式餐厅可以降级为小吃、轻食或低预算餐饮；
5. 用户带爸妈时，优先选择家庭友好、低等待、有座位、少走路的餐厅；
6. 用户说餐厅排队太久时，优先只替换餐饮点，不改变前面路线。

---

### 11.7 第二步：确定路线骨架

系统根据用户约束确定路线骨架。

#### 11.7.1 标准三小时路线

适用条件：

1. 用户可用时间约 2.5-4 小时；
2. 用户没有明确排除咖啡 / 餐饮；
3. 当前时间覆盖下午到晚饭前后。

骨架：

```text
起点 → 核心体验点 → 中段缓冲点 → 餐饮点
```

---

#### 11.7.2 时间压缩路线

适用条件：

1. 用户只剩 2 小时左右；
2. 用户明确赶时间；
3. 当前路线超出时间上限。

骨架：

```text
起点 → 核心体验点 → 餐饮点
```

处理：

1. 删除咖啡、休息、小吃、拍照增强等非必要节点；
2. 缩短核心体验点停留时间；
3. 保留基本路线闭环；
4. 总时长必须压到用户时间上限内。

---

#### 11.7.3 低预算路线

适用条件：

1. 用户降低预算；
2. 用户明确预算上限较低；
3. 当前路线超预算。

骨架：

```text
起点 → 核心体验点 → 免费 / 低消费缓冲点 → 低预算餐饮点
```

处理：

1. 免费景点优先；
2. 高消费咖啡降权或替换；
3. 高预算餐厅过滤或替换；
4. 主路线必须不超过预算上限；
5. 轻微超预算方案可作为后续备选，不作为 V1 主推荐。

---

#### 11.7.4 拍照增强路线

适用条件：

1. 用户明确想拍照；
2. 用户想更出片；
3. 用户第一次来杭州想留纪念。

骨架：

```text
起点 → 核心体验点 → 拍照增强点 → 缓冲 / 餐饮点
```

处理：

1. 提高 photoScore 和湖景标签权重；
2. 如果默认核心体验点已经适合拍照，可以不新增节点；
3. 如果新增拍照点导致明显超时或绕路，则用拍照点替换缓冲点；
4. 拍照增强不能压过时间和路线可执行性。

---

#### 11.7.5 安静 / 少游客路线

适用条件：

1. 用户不要游客太多；
2. 用户想安静一点；
3. 用户不想网红打卡。

骨架：

```text
起点 → 低拥挤湖景体验点 → 安静休息点 / 餐饮点
```

处理：

1. 降低热门核心点权重；
2. 提高 quietScore；
3. 提高低拥挤湖景点权重；
4. 不取消西湖体验，只替换为更分散的湖景体验；
5. 不跨出首期区域去过远景点。

---

### 11.8 ⭐ 第三步：候选 POI 选择

每个节点类型先取候选池，不直接写死地点。

候选池来源不是单一字段，而是来自四层：

```text
高德 POI 搜索结果
  ↓
后端标准化后的 poiCatalog
  ↓
后端内部 experienceTags / score 字段
  ↓
当前 mock 兜底数据
```

说明：

1. 高德 API 提供真实 POI 基础信息，例如地点名称、地址、经纬度、类型、距离等。
2. 后端将高德 POI 标准化为 `poiCatalog`。
3. `experienceTags` 和各类 score 字段是后端内部算法字段，不是高德原始字段，也不是前端必须展示字段。
4. 前端最终只展示被选中的 `places + route + diff`。
5. 如果高德 API 失败或候选不足，可以使用当前 mock 数据兜底。

每类节点建议至少保留 Top 3 候选。

候选筛选不直接依赖高德原始类型，而是基于后端标准化后的 `type`、`experienceTags` 和评分字段。

候选选择规则：

1. 不要每个策略都固定返回同一个点。
2. 同一类型候选先取 Top 3，再结合路线顺路程度选择。
3. 如果候选分数接近，优先选更顺路、风险更低、解释更清楚的点。
4. 如果某个 POI 已经在当前路线中，局部再规划时除非必要，不重复把它作为新增点。
5. 如果没有足够真实候选，使用 mock 兜底数据，但代码层面仍需保留候选池结构。
6. 最终返回前端前，需要把内部 `poiCatalog` 字段转换成前端当前可展示的 `places` 结构。

---

### 11.9 第四步：路线组合

系统不能只把单点最高分 POI 拼起来。

组合路线时需要计算：

1. 节点之间是否顺路；
2. 单段交通时间是否过长；
3. 总步行距离是否合理；
4. 停留时间是否合理；
5. 是否会超出用户总时间；
6. 是否覆盖用户主要需求；
7. 是否出现功能重复节点；
8. 是否违反预算或排除项。

路线组合优先级：

```text
满足硬约束
  > 顺路
  > 总时长合理
  > 类型完整
  > 风险较低
  > 偏好匹配
  > 亮点加分
```

---

### 11.10 路线总指标计算

每条候选路线需要计算总指标。

必须输出：

---

#### 11.10.1 总时长

总时长由三部分组成：

```text
总时长 = 各 POI 停留时间之和 + 各段交通时间之和 + 必要缓冲时间
```

---

#### 11.10.2 总预算

总预算由消费型节点估算：

```text
总预算 = 餐饮人均 + 咖啡 / 茶饮 / 小吃人均 + 其他消费节点
```

免费景点不计入预算。

---

#### 11.10.3 总等待风险

整体等待风险取路线中消费节点和热门节点的最高风险，并结合数量调整。

示例：

1. 所有节点 low → 整体 low；
2. 存在一个 medium → 整体 low-medium 或 medium；
3. 存在 high → 默认不进入主推荐，除非用户明确接受热门排队。

---

#### 11.10.4 步行距离

步行距离由各段交通距离累加。

第一版可以用 mock 或高德距离估计。

后续接高德路径规划后，可用真实步行路线距离替换。

---

### 11.11 ⭐ （待定）时间轴生成规则

时间轴必须让用户知道每个点几点到、几点走。

生成方式：

```text
当前节点到达时间
+ 当前节点停留时间
+ 下一段交通时间
= 下一节点到达时间
```

规则：

1. 起点停留时间可以设置为 5 分钟；
2. 核心体验点默认停留 25-40 分钟；
3. 咖啡 / 休息点默认停留 25-40 分钟；
4. 拍照增强点默认停留 15-25 分钟；
5. 餐饮点默认停留 45-60 分钟；
6. 时间压缩策略下，非餐饮节点停留时间可以缩短；
7. 如果结束时间超过用户限制，需要删节点或换更近点，而不是只压缩文案。

---

### 11.12 交通方式规则

第一版交通方式采用步行优先。

规则：

1. 西湖湖滨周边路线默认步行；
2. 每段交通信息通过 `transportSegments` 返回；
3. V1 不做公交换乘；
4. V1 不做自动混合交通方式最优解；
5. 如果用户带爸妈、少走路、时间紧，可以在 `transportSummary` 中提示部分路段可打车；
6. 后续可扩展为每段独立判断步行 / 打车 / 骑行。

第一版保留扩展能力，但不把混合交通作为本周主交付。

---

### 11.13 主推荐路线选择

系统最终只输出一条主推荐路线。

选择规则：

1. 必须满足硬约束；
2. 总分最高；
3. 如果总分接近，选更稳妥的路线；
4. 如果稳妥程度接近，选更顺路的路线；
5. 如果顺路程度接近，选更容易解释的路线；
6. 不为了亮点牺牲可执行性。

当两条路线差异小于一定阈值时，优先选择：

```text
改动更少
风险更低
步行更少
时间更稳
解释更清楚
```

## ⭐ 局部再规划规则

### 12.1 局部再规划总原则

局部再规划的核心原则是：

```text
最小必要改动
```

用户现场提出变化时，系统不应默认整条路线重算。

优先级：

1. 只改用户点名的节点；
2. 如果无法满足，再调整相邻节点；
3. 如果仍无法满足，再压缩或重排整条路线；
4. 只有路线整体不可行时，才重新生成完整路线。

示例：

用户说“餐厅排队太久”，系统应该优先替换餐厅，而不是把景点、咖啡、路线顺序全部换掉。

---

### 12.2 局部再规划流程

流程如下：

```text
接收当前路线 currentRoute
  ↓
识别 adjustmentType
  ↓
定位受影响节点
  ↓
确定硬约束 / 风险约束 / 软偏好变化
  ↓
在候选池中寻找替代 POI
  ↓
重新计算 route 指标
  ↓
生成完整新 route
  ↓
生成 diff
  ↓
返回给前端
```

第一版调整接口返回：

```text
route + diff
```

不要只返回局部 patch。

---

## 策略化方案版本说明

### 15.1 为什么 V1 不做三条并列路线

原始需求中包含策略化方案，例如：

1. 少走路；
2. 低等待；
3. 低预算；
4. 体验更完整；
5. 更适合拍照。

但策略化方案不一定要表现为“三条并列路线卡片”。

本产品的核心目标是降低用户决策成本，而不是把多个方案扔给用户继续比较。

因此，策略化方案应优先采用以下形态：

```text
默认给出一条主推荐路线
用户有新偏好或现场变化时，再进行策略化调整
```

也就是说，系统先给用户一个综合最稳妥的路线，让用户可以直接出发。

只有当用户表达“少走路”“不想排队”“预算降低”“想拍照”“只剩 2 小时”等变化时，系统再切换策略并返回新路线。

这比一开始展示多条路线更符合“现在就出发”的场景。

---

### 15.2 V1 策略化方案形态

V1 的策略化方案形态是：

```text
一条主推荐路线 + 策略化调整入口
```

也就是：

1. 默认先展示一条主推荐路线；
2. 用户点击“餐厅排队太久”“预算降到100”“不要咖啡”“只剩2小时”“想更适合拍照”等按钮；
3. 后端根据 `adjustmentType` 返回新的完整 `route`；
4. 前端展示新路线和 `diff`；
5. 用户能看到系统如何根据策略改变路线。

这样做的原因：

1. 当前前端已经实现主路线和调整按钮；
2. 当前后端 mock API 已按 `route + diff` 设计；
3. 比赛周期紧，优先保证闭环稳定；
4. 动态再规划比一次性展示多条路线更能体现 Agent 能力；
5. 多路线并列展示会新增前端组件、API 字段和选择逻辑，容易影响本周交付；
6. 用户在“现在就出发”场景下更需要一个可执行主方案，而不是更多选择负担。

因此，V1 不是不做策略化方案，而是把策略化方案收敛到“主路线后的策略调整”。

---

### 15.3 后续策略化交互方向

后续不把“三条并列路线卡片”作为默认演进方向，而是优先演进为：

```text
自然语言策略切换 + 新旧方案对比 + 必要时局部方案比较
```

例如：

用户输入：

```text
这条能不能少走一点？
```

系统返回：

```text
已将路线调整为少走路版本，删除中途绕行点，步行从 2.6km 降到 1.8km，但拍照点减少 1 个。
```

用户输入：

```text
如果不去咖啡店，时间能省多少？
```

系统返回：

```text
可以省约 35 分钟，人均从 120 元降到 78 元，但中途坐下休息的时间会减少。
```

用户输入：

```text
我想拍照，但又不想走太远，帮我权衡一下。
```

系统返回：

```text
建议保留断桥，暂不新增第二个拍照点。这样能保留西湖经典视角，同时避免步行距离明显增加。
```

后续策略化能力的重点不是“展示三条路线让用户选”，而是：

1. 用户用自然语言提出取舍；
2. 系统识别用户当前更看重什么；
3. 后端规则引擎重新调整权重；
4. 返回一条新的主推荐路线；
5. 用 diff 解释新旧方案变化；
6. 必要时只比较一个关键取舍，而不是展示大量备选。

---

### 15.4 多路线并列卡片是否彻底不做

多路线并列卡片不是完全不做，但不作为默认主路径。

它可以作为以下场景的可选能力：

1. 比赛演示时需要快速展示系统能生成不同策略；
2. 用户明确说“给我比较一下几个方案”；
3. 用户在结果页主动点击“查看其他策略”；
4. 后续 A/B 测试发现用户确实需要并列比较。

如果要做，多路线卡片应作为“用户主动展开”的辅助功能，而不是默认打断主推荐。

建议字段可以保留为后续扩展：

```JSON
{
  "strategyComparisons": [
    {
      "strategy": "recommended",
      "label": "综合推荐",
      "summary": "最稳妥，时间和步行较均衡",
      "keyTradeoff": "没有极致优化拍照或预算",
      "route": {}
    },
    {
      "strategy": "photo",
      "label": "更适合拍照",
      "summary": "增加湖景拍照点",
      "keyTradeoff": "步行略增加",
      "route": {}
    }
  ]
}
```

但这不是 V1 本周主交付。

## 兜底与异常处理

### 16.1 找不到完全满足路线

真实产品中，规则可能找不到完美路线。此时不能硬编。

处理：

1. 保留硬约束；
2. 放宽软偏好；
3. 给出最接近方案；
4. 明确说明哪个偏好没有完全满足。

示例：

```text
当前预算和时间都较紧，系统优先保留断桥和晚餐，暂不安排咖啡点。
```

---

### 16.2 高德 API 无结果或失败

处理：

1. 使用本地 `poiCatalog` 缓存；
2. 如果缓存也不足，使用 mock 兜底路线；
3. 前端仍返回可展示 route，不让页面空掉；
4. `meta` 中可标记 mock 或 fallback。

---

### 16.3 用户需求冲突

示例：

```text
用户想少走路，又想多拍几个点。
```

处理：

1. 时间和距离优先；
2. 降低节点数量；
3. 保留最核心拍照点；
4. 在解释中说明取舍。

输出示例：

```text
为了减少步行，系统没有继续增加第二个拍照点，而是保留了距离更近的湖景点。
```

## 为什么第一版使用规则算法

第一版使用规则算法，而不是复杂机器学习模型，原因如下：

1. 当前没有足够真实用户行为数据；
2. 当前没有路线满意度标注；
3. 黑客松周期短，复杂模型不可控；
4. 路线规划需要强可解释性；
5. 规则算法更适合冷启动阶段；
6. 后端可以快速实现并调试；
7. 产品负责人可以明确控制路线取舍。

第一版规则算法不是低级方案，而是冷启动阶段更合理的落地方案。

后续当用户数据增加后，可以逐步演进为：

```text
V1：专家规则 + 高德 POI + 自建标签 + 固定权重
V1.5：基于用户点击和调整行为手动调权
V2：自然语言策略切换和局部方案比较
V3：尝试排序模型和个性化推荐
```

## 为什么不让 LLM 直接生成路线

LLM 适合处理语言，但不适合直接承担路线决策。

不让 LLM 直接生成完整路线的原因：

1. LLM 可能编造 POI；
2. LLM 不稳定掌握实时营业时间；
3. LLM 不可靠掌握真实距离；
4. LLM 不知道实时排队和拥挤情况；
5. LLM 生成路线难以保证顺路；
6. LLM 难以稳定处理预算和时间硬约束；
7. LLM 难以解释每次局部再规划的具体指标变化。

正确做法是：

```text
LLM 解析用户需求
规则引擎生成路线
LLM 解释推荐理由
```

这样既利用了 LLM 的自然语言能力，又保证路线决策可控。

## 为什么先做西湖周边

第一版选择杭州西湖周边，不是因为场景小，而是为了把一个高频、清晰、可验证的即时出行场景做深。

原因：

1. 西湖周边本地出行需求高频；
2. 湖滨、断桥、白堤、北山街等区域适合 2-4 小时短路线；
3. 场景天然包含景点、拍照、咖啡、餐饮、休息点；
4. POI 密度高，适合做多目标路线组合；
5. 用户临时调整需求丰富，例如排队、预算、时间不足、想拍照；
6. 小范围更容易保证数据质量和推荐可执行性。

第一版不是做全城攻略，而是验证：

```text
用户一句话需求
→ 结构化约束
→ 真实 POI
→ 多维评分
→ 可执行路线
→ 局部再规划
→ 新旧方案对比
```

这个闭环跑通后，再扩展到更多区域。

## 面试可讲版本

这个项目的核心不是“用 GPT 写旅游攻略”，而是做一个可执行的本地路线决策系统。

可以这样总结：

```text
我们把 LLM 放在理解和解释层，
把高德放在真实 POI 和地图数据层，
把后端规则引擎放在路线决策层，
把前端放在结构化展示和交互层。
```

第一版选择规则算法，是因为项目处于冷启动阶段，缺少用户行为数据和训练样本。

规则算法能保证：

1. 可解释；
2. 可控；
3. 可快速落地；
4. 便于前后端联调；
5. 便于后续根据用户反馈调权。

第一版不默认展示多条路线，是因为产品目标是降低用户决策成本。

系统应该先给出一条可执行主方案，再在用户提出新约束时进行策略化调整。

第一版选择西湖湖滨周边，是因为这个场景边界清晰、POI 密度高、用户调整需求真实，适合验证：

```text
自然语言约束
→ 真实 POI
→ 多维评分
→ 路线生成
→ 动态再规划
```

后续上线后，可以通过用户点击、删除、替换、保存、分享等行为数据，逐步从固定规则演进到动态权重和个性化推荐。

---

## 一句话总结

第一版路线规划系统的核心不是“生成一篇好看的攻略”，而是：

```text
用 LLM 理解用户，
用高德提供真实地点，
用后端规则引擎做路线决策，
用前端展示可执行路线和调整变化。
```

V1 先用规则算法完成冷启动，保证路线可执行、可解释、可调整。

后续通过真实 POI 扩展、用户行为反馈、策略调权和自然语言再规划，逐步演进为更智能的本地路线决策 Agent。
